멀티에이전트 방법론 가이드
서브에이전트·멀티에이전트가 어떻게 작동하는지, Git Worktree 기반 병렬 개발 방법론까지 한눈에 정리합니다.
01 — 핵심 개념
서브태스크 · 서브에이전트 · 오케스트레이터
멀티에이전트 시스템은 세 개의 개념이 맞물려 작동합니다. 역할이 명확히 구분될수록 시스템은 더 예측 가능하고 안정적으로 동작합니다.
🗂
서브태스크
큰 작업을 독립적으로 실행 가능한 단위로 쪼갠 것. 각 서브태스크는 명확한 입력과 출력을 가져야 합니다.
🤖
서브에이전트
서브태스크를 실행하는 주체. 독립적인 시스템 프롬프트(역할)를 가진 Claude 인스턴스. 자신의 태스크만 알고 전체 그림은 모릅니다.
🧠
오케스트레이터
전체 작업 흐름을 설계하고, 서브에이전트에게 태스크를 분배하며, 결과를 수집·통합하는 중앙 에이전트.
🔁
컨텍스트 전달
Claude는 호출 간 메모리가 없습니다. 이전 에이전트 결과를 다음 호출의 프롬프트에 명시적으로 포함시켜야 연결됩니다.
02 — 작동 흐름
멀티에이전트 파이프라인
오케스트레이터가 태스크를 분해하고, 서브에이전트들이 병렬 또는 순차 실행하며, 오케스트레이터가 결과를 통합합니다.

03 — 의사결정
단일 에이전트 vs 멀티에이전트
멀티에이전트가 항상 더 좋은 것은 아닙니다. 작업의 특성에 맞게 선택하는 것이 핵심입니다.
기준 단일 에이전트 멀티에이전트
작업 규모소~중형, 명확한 목표대형, 복잡한 목표
병렬 가능성낮음 — 순차 의존높음 — 동시 실행 가능
전문성 요구단일 역할로 처리 가능역할별 전문화 필요
컨텍스트 크기단일 윈도우 내 처리 가능문서가 너무 많아 분산 필요
오류 복구단순, 전체 재시도복잡, 부분 재시도 가능
비용 · 속도낮음 · 빠름높음 · 조율 오버헤드 존재
구현 복잡도단순복잡 (오케스트레이션 필요)
Anthropic 권장 원칙
단순한 작업에는 단일 Claude 호출이 충분합니다. 멀티에이전트는 병렬화가 명확히 가능하고, 컨텍스트가 단일 윈도우를 초과하거나, 역할별 전문화가 필요할 때만 도입하세요. 에이전트 수는 항상 최소로 유지하세요.
04 — Git 방법론
Git Worktree 기반 병렬 개발
멀티에이전트가 각각 독립된 브랜치에서 동시에 작업하도록 Git Worktree를 활용하면 충돌 없이 병렬 개발이 가능합니다.
Git Worktree 개념
하나의 Git 저장소에서 여러 브랜치를 각각 다른 디렉토리에 동시에 체크아웃하는 기능. 에이전트마다 별도 작업 디렉토리를 부여해 충돌 없이 병렬 작업이 가능합니다.
# 에이전트 A용 워크트리 생성
git worktree add ../agent-a feat/research
# 에이전트 B용 워크트리 생성
git worktree add ../agent-b feat/analysis
# 워크트리 목록 확인
git worktree list
병렬 개발 흐름 에이전트 협업
오케스트레이터가 브랜치 전략을 설계하고, 각 서브에이전트는 전용 워크트리에서 독립 실행됩니다. 완료 후 PR → 코드 리뷰 → 메인 브랜치 병합.
# 오케스트레이터가 전략 수립
main
├── feat/research ← 에이전트 A
├── feat/analysis ← 에이전트 B
└── feat/report ← 에이전트 C
# 완료 후 병합
git merge feat/research feat/analysis
에이전트 팀 단계별 워크플로
1
오케스트레이터 — 태스크 설계
전체 작업을 서브태스크로 분해하고, 각 에이전트의 역할·입출력·의존관계를 정의합니다. Git 브랜치 전략도 이 단계에서 결정합니다.
2
서브에이전트 — 병렬 실행
각 에이전트가 전용 워크트리(브랜치)에서 독립적으로 서브태스크를 실행합니다. Claude Code를 사용하면 에이전트가 파일 읽기·코드 실행·웹 검색까지 자율 처리합니다.
3
검증 · 통합
오케스트레이터가 각 에이전트 결과를 검토하고, 품질 기준 미달 시 해당 에이전트만 재실행합니다. 통과된 결과는 메인 브랜치로 병합합니다.
4
배포 · 자동화
GitHub Actions 등 CI/CD와 연결하면 에이전트 파이프라인이 자동 트리거됩니다. n8n, LangGraph 등 오케스트레이션 도구와 결합하면 완전 자동화가 가능합니다.
Git Worktree × 에이전트 팀 타임라인
각 에이전트가 독립 브랜치에서 동시에 커밋 → PR로 main에 통합. agent/writer는 작업 진행 중 (점선). 에이전트 간 파일 충돌 없음.
