# 멀티에이전트 도입 의사결정 프레임워크

## 1. 핵심 원칙

> Anthropic 권장: **단순한 작업에는 단일 에이전트가 항상 우선**
> 멀티에이전트는 명확한 이득이 있을 때만 도입하며, 에이전트 수는 최소로 유지

---

## 2. 도입 적합 조건 (3개 이상 충족 시 권장)

| # | 조건 | 판단 질문 | 가중치 |
|---|------|----------|--------|
| 1 | **병렬화 가능성** | 독립적으로 동시 실행 가능한 서브태스크가 있는가? | ⭐⭐⭐ (가장 중요) |
| 2 | **컨텍스트 오버플로** | 단일 윈도우(~20만 토큰)로 처리 불가한 규모인가? | ⭐⭐ |
| 3 | **역할 전문화** | 서로 다른 전문성이 동시에 필요한가? | ⭐⭐ |
| 4 | **반복·자동화** | 파이프라인화하여 반복 실행할 계획인가? | ⭐⭐ |
| 5 | **부분 실패 허용** | 일부 서브태스크 실패 시 나머지는 유효한가? | ⭐ |

### 점수 해석 가이드

```
종합 점수 = 복잡도×0.35 + 병렬가능성×0.45 + 오버헤드적정성×0.20

≥ 70점  → 멀티에이전트 강력 권장
50~69점 → 조건부 권장 (규모·팀 역량 고려)
< 50점  → 단일 에이전트 권장
```

**병렬 가능성(0.45)이 가중치 가장 높음:**
- 병렬화가 불가하면 멀티에이전트의 핵심 이점(속도) 없음
- 순차 의존 작업을 멀티에이전트화하면 오버헤드만 증가

## 3. 도입 부적합 조건 (하나라도 해당 시 단일 에이전트 권장)

| 조건 | 이유 |
|------|------|
| A → B → C 완전 순차 의존 | 병렬화 불가, 오케스트레이션 비용만 증가 |
| 단순 요약·번역·분류 | 단일 호출로 충분, 복잡도 과잉 |
| 빠른 응답 필요 (< 3초) | 에이전트 조율 레이턴시 발생 |
| 팀이 오케스트레이션 유지 불가 | 운영 비용이 이점 초과 |
| 작업이 매번 달라 재사용 불가 | 파이프라인화 이점 없음 |

## 4. 실전 사용 사례별 판단

### ✅ 멀티에이전트 적합

| 사용 사례 | 병렬 구조 | 이유 |
|----------|----------|------|
| 다국어 콘텐츠 동시 생성 | A·B·C 국가 병렬 | 완전 독립 |
| 대규모 코드 리팩토링 | 모듈별 병렬 | Git Worktree 활용 |
| 멀티소스 리서치 보고서 | 소스별 병렬 수집 | 컨텍스트 오버플로 |
| CI/CD 자동화 파이프라인 | 빌드·테스트·배포 병렬 | 반복 자동화 |
| 대규모 데이터 처리 | 청크 분할 병렬 | 규모 문제 |

### ❌ 단일 에이전트 적합

| 사용 사례 | 이유 |
|----------|------|
| 챗봇, 대화 인터페이스 | 순차 대화 흐름, 빠른 응답 필요 |
| 단일 문서 요약 | 단일 호출로 충분 |
| 이미지 하나 분석 | 컨텍스트 충분 |
| 간단한 코드 디버깅 | 단일 역할 |
| 정형화된 데이터 추출 | 단순 작업 |

## 5. 비용·복잡도 트레이드오프

```
멀티에이전트 도입 시 추가 비용:

1. API 비용 증가
   - 오케스트레이터 + N개 서브에이전트 = N+1배 호출
   - 컨텍스트 전달로 입력 토큰 중복 발생

2. 개발 복잡도
   - 오케스트레이터 로직 설계 필요
   - 에이전트 간 인터페이스 정의
   - 에러 핸들링 + 재시도 로직

3. 운영 복잡도
   - 각 에이전트 프롬프트 개별 관리
   - 부분 실패 시 디버깅 난이도 증가
   - 레이턴시 증가 (조율 오버헤드)

이득 > 비용이 되려면:
- 병렬화로 절약되는 시간이 조율 오버헤드 > 시간
- 또는 단일 호출로 불가능한 규모의 작업
```

## 6. 안티패턴

| 안티패턴 | 증상 | 해결책 |
|---------|------|--------|
| 과분할 | 5줄짜리 작업을 3개 에이전트로 | 단일 에이전트로 통합 |
| 상태 공유 가정 | 에이전트 B가 A의 내부 상태 "알고 있다고" 가정 | 컨텍스트 명시적 전달 |
| 에러 무시 | 서브에이전트 실패 시 빈 결과로 진행 | 체크포인트 + 재시도 로직 |
| 병렬화 불가 작업 병렬화 | 순차 의존 작업에 병렬 구조 적용 | 의존성 분석 후 DAG 설계 |
| 오케스트레이터 과부하 | 오케스트레이터가 실행도 담당 | 역할 분리 철저히 |
