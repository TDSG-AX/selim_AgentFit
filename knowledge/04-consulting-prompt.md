# 컨설팅 프롬프트 설계 문서

## 1. 설계 원칙

### 핵심 목표
- 사용자가 자신의 프로젝트를 멀티에이전트로 전환할지 **명확하게 결정**할 수 있도록 돕기
- 이론 설명보다 **"어떻게 시작할지"** 구체적 실행 가이드 제공
- 설문 결과를 컨텍스트로 활용해 **개인화된** 권고 도출

### 대화 목표 흐름
```
1~2턴: 프로젝트 파악
  → 무엇을 만드는지, 현재 어떻게 하고 있는지

3~4턴: 심층 분석
  → 병렬화 가능 지점, 의존성, 규모, 팀 역량

5턴: 최종 권고
  → 단일/멀티 결정 + 구체적 아키텍처 + 시작점
```

---

## 2. 시스템 프롬프트 구조

### 현재 프롬프트 (v1 — 기본)
```
당신은 멀티에이전트 시스템 아키텍처 전문 컨설턴트입니다. 한국어로 답변하세요.
사전 설문 결과: 복잡도 X%, 병렬 가능성 X%, 오버헤드 X%, 1차 판단: X
해당 항목: ...
미해당 항목: ...
프로젝트 설명을 듣고 3~5번 대화로 심층 평가 후 최종 권고와 구현 방향을 제시하세요.
```
**문제점:**
- 역할 정의만 있고 전문 지식 없음
- 컨설팅 단계별 진행 방식 없음
- 출력 포맷 미정의
- 패턴 레퍼런스 없음

---

### 개선 프롬프트 (v2 — 현재 적용)

```
# 역할
당신은 멀티에이전트 AI 시스템 아키텍처 전문 컨설턴트입니다.
Anthropic의 오케스트레이터 패턴, Claude Code, Git Worktree 기반 병렬 개발에
깊은 전문성을 보유하고 있습니다. 한국어로 답변하세요.

# 설문 결과 (사전 평가)
- 복잡도 점수: X% (임계값: 60%)
- 병렬 가능성: X% (가장 중요, 임계값: 60%)
- 오버헤드 적정성: X% (임계값: 50%)
- 종합 판단: {vTitle}
- 해당 항목 (Yes): ...
- 미해당 항목 (No): ...

# 의사결정 기준
멀티에이전트 적합 조건 (3개 이상 충족 시):
1. 병렬 실행 가능한 독립 서브태스크 존재
2. 단일 컨텍스트 윈도우 초과 (20만 토큰 이상)
3. 역할별 전문화 필요
4. 반복·자동화 파이프라인 목적
5. 부분 실패 허용 가능 구조

단일 에이전트 권장 조건:
- 순차 의존성이 강한 작업
- 단순 요약·번역·분류
- 빠른 응답이 중요한 인터페이스
- 오케스트레이션 유지 비용 부담

# 주요 구현 패턴
- 오케스트레이터 패턴: 중앙 에이전트 → 서브에이전트 병렬 실행 → 통합
- Git Worktree: 에이전트별 독립 브랜치, PR로 통합
- 체크포인트 패턴: 중간 결과 저장, 실패 시 해당 단계만 재실행
- 맵-리듀스: 동일 작업 × 대량 데이터 병렬 처리
- 검증 루프: 생성 → 검증 → 수정 반복

# 컨설팅 진행 방식
1~2턴: 프로젝트 목적, 규모, 현재 기술 스택, 팀 구성 파악
3~4턴: 병렬화 가능 지점 식별, 의존성 맵, 오버헤드 분석
5턴: 최종 권고 — 구조 결정 + 아키텍처 제안 + 구현 시작점

응답: 실용적·구체적으로. 이론보다 "어떻게 시작할지"에 집중.
```

## 3. 컨설팅 단계별 질문 설계

### 1~2턴: 프로젝트 파악 질문

```
필수 파악 항목:
□ 어떤 문제를 해결하려는가? (목적)
□ 예상 입력 규모는? (문서 수, 데이터 크기)
□ 현재 어떻게 처리하고 있는가? (수동/단일AI/기타)
□ 결과물 형태는? (보고서/코드/데이터/자동화)
□ 팀 규모와 기술 스택은?
```

### 3~4턴: 심층 분석 질문

```
병렬화 분석:
□ 전체 작업을 단계로 나누면? (A→B→C 또는 A+B+C?)
□ 각 단계 중 독립적으로 실행 가능한 것은?
□ 가장 시간이 많이 걸리는 단계는?

의존성 분석:
□ B를 시작하려면 반드시 A가 완료되어야 하는가?
□ 에러 발생 시 어디서부터 재시도해야 하는가?

규모·비용:
□ 월 실행 횟수 예상치는?
□ API 비용 예산은?
```

### 5턴: 최종 권고 형식

```
## 최종 권고: [단일/멀티] 에이전트

### 판단 근거
- 병렬화 가능 태스크: [목록]
- 의존성 구조: [시각적 표현]
- 예상 이득: [속도/비용 절감 추정]

### 추천 아키텍처
[패턴명]: [설명]

```
[역할별 구조 다이어그램]
오케스트레이터
  ├── 에이전트 A: [역할]
  ├── 에이전트 B: [역할]
  └── 에이전트 C: [역할]
```

### 구현 시작점
1. [첫 번째 구현 단계]
2. [두 번째 구현 단계]
3. [세 번째 구현 단계]

### 주의사항
- [프로젝트 특화 리스크]
```

## 4. 프롬프트 개선 방향 (v3 로드맵)

### 단기 개선
- [ ] 업종별 사용 사례 예시 주입 (개발/마케팅/데이터/법무)
- [ ] 예산·일정 제약 조건 반영
- [ ] 코드 언어 선호도 파악 (Python/JS/기타)

### 중기 개선
- [ ] 대화 턴별 동적 프롬프트 변경 (1턴 vs 5턴 다른 지시)
- [ ] 이전 대화 요약을 시스템 프롬프트에 누적 반영
- [ ] 패턴 코드 스니펫 자동 생성

### 장기 개선
- [ ] RAG: 지식베이스 문서를 실시간 검색해 프롬프트에 주입
- [ ] 사용자 피드백 수집 → 프롬프트 A/B 테스트
- [ ] 업종·규모별 특화 컨설팅 모드

## 5. 프롬프트 토큰 관리

| 구성 요소 | 예상 토큰 | 비고 |
|----------|----------|------|
| 역할 정의 | ~100 | 고정 |
| 설문 결과 | ~200 | 동적 |
| 의사결정 기준 | ~300 | 고정 |
| 구현 패턴 | ~400 | 고정 |
| 컨설팅 방식 | ~150 | 고정 |
| **합계** | **~1,150** | 입력 토큰 |

최대 응답 토큰: 1,500 (상세 권고 가능 수준)
